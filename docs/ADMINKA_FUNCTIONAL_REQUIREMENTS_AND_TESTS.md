# `adminka`: функциональные требования, методы, сценарии и тесты по разделам

## Назначение документа

Этот файл фиксирует:
- функциональные требования по каждому разделу;
- используемые API-методы;
- ключевые пользовательские сценарии;
- последовательности тестирования и ожидаемые результаты.

Обязательное правило:
- при изменении логики любого раздела обновлять соответствующий подраздел этого файла в том же PR/коммите.

---

## 1) Раздел «Сотрудники»

### 1.1 Функциональные требования
- Пагинация, сортировка, фильтрация списка сотрудников.
- Быстрый переход к странице.
- Открытие карточки сотрудника по клику строки.
- Редактирование основных полей карточки.
- Отображение/редактирование связанных данных (должности, подчинения, связи) по текущим правилам UI.
- Ссылки на карточку организации из полей «Организация».

### 1.2 Используемые методы
- `POST /api/admin/employees`
- `POST /api/admin/list_employees`
- `POST /api/admin/list_positions`
- `POST /api/admin/list_organizations`
- (прочие методы карточки сотрудника по текущей реализации backend `ApiController`)

### 1.3 Сценарии использования
- Пользователь открывает раздел, фильтрует сотрудников по ФИО/статусу/организации.
- Пользователь кликает строку сотрудника и открывает карточку.
- Пользователь редактирует карточку и сохраняет.
- Пользователь переходит в связанные таблицы и проверяет ссылки на организацию.

### 1.4 Тесты и ожидаемые результаты
- **Тест S1. Загрузка списка**
  - Шаги: открыть раздел «Сотрудники».
  - Ожидание: таблица загружается, ошибок API нет.
- **Тест S2. Фильтр**
  - Шаги: задать фильтр, применить.
  - Ожидание: таблица обновляется, показываются только подходящие записи.
- **Тест S3. Сортировка**
  - Шаги: клик по заголовку сортируемой колонки.
  - Ожидание: порядок строк меняется согласно направлению сортировки.
- **Тест S4. Карточка сотрудника**
  - Шаги: клик по строке сотрудника.
  - Ожидание: открывается карточка, URL синхронизируется, данные корректны.
- **Тест S5. Сохранение карточки**
  - Шаги: изменить поле, нажать «Сохранить».
  - Ожидание: запись сохраняется, после перезагрузки значение остается.

## 2) Раздел «Организации» (список + карточка)

### 2.1 Функциональные требования
- Список организаций:
  - фильтры, сортировка, пагинация, экспорт;
  - фильтр по типам организаций (multi-select, чекбоксы);
  - исключение строк по условию backend (валидный `sap_id`, `parent_id` логика по текущим правилам).
- Карточка организации:
  - отдельный URL;
  - вкладка «Основная информация»;
  - редактирование полей (включая `address`, `country`, `resident`, `organUnitTypes`, `additional`-поля);
  - удаление с проверкой зависимостей.

### 2.2 Используемые методы
- `POST /api/admin/organizations`
- `POST /api/admin/list_organizations`
- `POST /api/admin/list_organization_unit_types`
- `POST /api/admin/list_countries`
- `GET /api/admin/organization/{organUnitId}`
- `PATCH /api/admin/organization/{organUnitId}`
- `DELETE /api/admin/organization/{organUnitId}`

### 2.3 Сценарии использования
- Пользователь открывает список организаций, использует фильтр по типу.
- Пользователь открывает карточку организации по строке.
- Пользователь редактирует основную информацию и сохраняет.
- Пользователь удаляет организацию, получает либо успешное удаление, либо ошибку с таблицей-источником ссылки.

### 2.4 Тесты и ожидаемые результаты
- **Тест O1. Фильтр по типам**
  - Шаги: выбрать 2+ типов в фильтре.
  - Ожидание: выдача соответствует выбранным типам.
- **Тест O2. Открытие карточки**
  - Шаги: клик по строке организации.
  - Ожидание: открывается карточка с корректным URL.
- **Тест O3. Сохранение адреса**
  - Шаги: изменить адрес, сохранить, обновить страницу.
  - Ожидание: адрес сохранен в `party.address.full_address`.
- **Тест O4. Сохранение типа организации**
  - Шаги: изменить набор типов, сохранить.
  - Ожидание: связи в `party.organ_unit_organ_unit_types` обновлены.
- **Тест O5. Удаление с зависимостями**
  - Шаги: попытаться удалить используемую организацию.
  - Ожидание: 409 + имя таблицы, где найдена ссылка.

## 3) Вкладка «ДаДата» в карточке организации

### 3.1 Функциональные требования
- Вкладка показывается только для резидента.
- Автозапрос выполняется только при открытии карточки, когда `data_info` пустой и организация резидент.
- Кнопка `Обновить` выполняет принудительное обновление.
- Данные хранятся в `party.organ_unit.data_info`.
- Отображение разделено на 2 блока:
  - «Информация о компании»;
  - «Адресная информация».
- Заголовки блоков поддерживают сворачивание/разворачивание.
- Поддержка:
  - человекочитаемых названий полей;
  - локализации статусов (`ACTIVE` и т.п.);
  - локализации `branch_type` и `type`;
  - форматирования date-полей.
- Сравнение `inn/kpp/ogrn` с основной карточкой и подсветка расхождений.
- Чекбоксы видимости параметров + режим «Только видимые атрибуты».
- Drag-n-drop внутри каждого блока с единым глобальным порядком для всех карточек.

### 3.2 Используемые методы
- `POST /api/admin/organization/{organUnitId}/dadata/refresh`
- `GET /api/admin/organization/{organUnitId}` (поле `data_info`)

### 3.3 Сценарии использования
- Пользователь открывает вкладку и смотрит текущий снимок `data_info`.
- Пользователь обновляет ДаДата по кнопке.
- Пользователь скрывает/показывает параметры чекбоксами.
- Пользователь меняет порядок параметров drag-n-drop.

### 3.4 Тесты и ожидаемые результаты
- **Тест D1. Условие автозапроса**
  - Шаги: открыть карточку резидента с пустым `data_info`.
  - Ожидание: автозапрос выполняется и наполняет `data_info`.
- **Тест D2. Нерезидент**
  - Шаги: открыть карточку нерезидента.
  - Ожидание: вкладка ДаДата не показывается, автозапрос не выполняется.
- **Тест D3. Ручное обновление**
  - Шаги: нажать `Обновить`.
  - Ожидание: новые данные сохранены в `data_info`, таблицы обновлены.
- **Тест D4. Локализация значений**
  - Шаги: проверить поля типа/статуса/подразделения.
  - Ожидание: отображаются русские значения.
- **Тест D5. Даты**
  - Шаги: проверить поля `*_date`/`*.date`.
  - Ожидание: формат `ДД.ММ.ГГГГ` или `ДД.ММ.ГГГГ ЧЧ:ММ:СС`.
- **Тест D6. Подсветка расхождений**
  - Шаги: задать отличающиеся `inn/kpp/ogrn` в карточке.
  - Ожидание: расхождения подсвечены в основной вкладке и в ДаДата.
- **Тест D7. Видимость атрибутов**
  - Шаги: снять чекбокс с параметра, обновить страницу.
  - Ожидание: параметр скрыт, настройка сохранена в `localStorage`.
- **Тест D8. Drag-n-drop порядок**
  - Шаги: изменить порядок в блоке, открыть другую карточку.
  - Ожидание: порядок тот же (глобально единый), между блоками перенос невозможен.
- **Тест D9. Автоскролл при drag**
  - Шаги: перетаскивать за верх/низ таблицы.
  - Ожидание: быстрый автоскролл вверх/вниз без инверсии направления.

## 4) Раздел «Связи сотрудников»

### 4.1 Функциональные требования
- Табличный список связей с фильтрами и сортировкой.
- Открытие ссылок на карточку организации в новой вкладке.
- Редактирование/добавление связи по текущим правам и UI.

### 4.2 Используемые методы
- `POST /api/admin/relations`
- `POST /api/admin/list_relations`
- `POST /api/admin/list_employees`
- `POST /api/admin/list_organizations`

### 4.3 Сценарии использования
- Поиск связи по сотруднику/организации.
- Переход на карточку организации из строки.
- Изменение связи и сохранение.

### 4.4 Тесты и ожидаемые результаты
- **Тест R1. Список связей**
  - Ожидание: данные загружаются, фильтрация работает.
- **Тест R2. Ссылка на организацию**
  - Ожидание: открывается новая вкладка с карточкой организации.
- **Тест R3. Сохранение изменений**
  - Ожидание: после сохранения и перезагрузки данные не теряются.

## 5) Раздел «Настройки отчетов»

### 5.1 Функциональные требования
- CRUD шаблонов отчетов.
- Настройка SQL, валидация SQL, предпросмотр результата.
- Генерация Excel.
- Настройки организаций/групп доступа.
- Поддержка логотипа шаблона (`report_logo`).

### 5.2 Используемые методы
- `POST /api/admin/report-templates`
- `POST /api/admin/report-template/sql/validate`
- `POST /api/admin/report-template/sql/results`
- `POST /api/admin/report-template/excel-preview`
- `POST /api/admin/report-template/excel`
- `GET/PATCH /api/admin/report-template/{reportTemplateId}`
- дополнительные методы add/delete организаций и access groups.

### 5.3 Сценарии использования
- Создание шаблона, ввод SQL, проверка корректности.
- Предпросмотр результата, экспорт в Excel.
- Привязка/отвязка организаций и групп доступа.

### 5.4 Тесты и ожидаемые результаты
- **Тест T1. Валидация SQL**
  - Ожидание: валидный SQL проходит, невалидный возвращает понятную ошибку.
- **Тест T2. Preview**
  - Ожидание: таблица предпросмотра заполняется.
- **Тест T3. Excel**
  - Ожидание: выгружается корректный файл.
- **Тест T4. Логотип**
  - Ожидание: загруженный логотип сохраняется и используется в экспорте.

- `POST /api/admin/print-form-template`

### 6.3 Сценарии использования

- Выполнить распознавание полей.
- Настроить mapping и параметры overlay.
- Сгенерировать документ.

### 6.4 Тесты и ожидаемые результаты
- **Тест P1. Создание шаблона**
  - Ожидание: шаблон создается и отображается в списке.
- **Тест P2. Распознавание**
  - Ожидание: placeholder-поля извлекаются корректно.
- **Тест P3. Генерация PDF**
  - Ожидание: итоговый файл содержит подставленные значения в корректных координатах.

## 7) Сквозные нефункциональные требования

- Производительность:
  - без заметных лагов при выборе строк, пагинации и фильтрах.
- UX:
  - предсказуемое поведение dropdown/combobox без flicker;
  - корректная работа drag-n-drop и автоскролла.
- Надежность:
  - ошибки API обрабатываются без падения UI;
  - частичный update не должен очищать не переданные поля.
- Сохранение пользовательских настроек:
  - сортировки/фильтры/видимость атрибутов/порядок параметров.

## 8) Регламент обновления документа

При любом изменении функциональности:
- обновить соответствующий раздел (требования, методы, сценарии, тесты);
- добавить новые API-методы и удалить устаревшие;
- актуализировать ожидаемые результаты тестов;
- зафиксировать изменение в том же PR/коммите, где реализована логика.
