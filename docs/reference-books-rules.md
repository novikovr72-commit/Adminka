# `reference_books.rules` (jsonb)

Правила формирования/проверки **связок полей** таблицы справочника (не одиночных колонок).

## Формат

- Корень — **массив** объектов-правил (может быть пустым `[]`).
- Каждый элемент:
  - **`rule`** — `"uniqueness"` или `"presence"` (регистр не важен при сохранении, в БД нормализуется в нижний регистр).
  - **`fields`** — массив из **не менее двух** объектов:
    - **`tableName`** — имя столбца целевой таблицы справочника (как в БД). Поле **`id`** в правилах не указывается.

Допустимо любое количество правил каждого типа.

## Семантика (для реализации проверок данных)

| `rule`        | Смысл |
|---------------|--------|
| `presence`    | Хотя бы одно из перечисленных полей должно иметь значение (для связки «или»). |
| `uniqueness`  | Кортеж значений по перечисленным полям должен быть уникален среди строк таблицы. |

Одиночные обязательность/уникальность задаются в **properties** полей (`Обязательное`, `Уникальное значение`), а не через `rules`.

## Пример

```json
[
  {
    "rule": "presence",
    "fields": [
      { "tableName": "status_before_id" },
      { "tableName": "status_after_id" }
    ]
  },
  {
    "rule": "uniqueness",
    "fields": [
      { "tableName": "object_type" },
      { "tableName": "claim_organization_id" },
      { "tableName": "status_before_id" },
      { "tableName": "function_name" },
      { "tableName": "status_after_id" }
    ]
  }
]
```

Валидация структуры и имён столбцов — на бэкенде при создании справочника и при `PATCH .../reference-books/{id}/rules`.
