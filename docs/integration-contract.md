# Integration Contract: Модуль Администрирование

Документ фиксирует обязательные правила разработки для дальнейшей интеграции `adminka` в более крупную систему.

## 1. Обязательная структуризация новых разделов

Каждый новый раздел backend реализуется отдельными классами по слоям:

- `...Controller` — только HTTP endpoints и маршрутизация;
- `...Service` — бизнес-логика и orchestration;
- `...Repository` — SQL, запросы к БД, транзакционные операции;
- `...Core` / `...Facade` — отдельные специализированные ядра (по необходимости);
- `dto` / `domain` / `mapper` — отдельные модели и преобразования (по необходимости).

## 2. Запрещенные практики

- Нельзя смешивать HTTP, бизнес-логику и SQL в одном классе.
- Нельзя добавлять новый функционал в общий монолитный класс вместо выделения нового доменного набора классов.
- Нельзя добавлять SQL в `controller` и `service`, если это DAO-операции.

## 3. Правило для всех следующих изменений

Любое новое дополнение/функциональный блок в `adminka` принимается только при соблюдении пунктов 1 и 2.

Это правило является частью интеграционного контракта с командой, которая встраивает модуль в целевую систему.
